Consistency is King**- Establish and **strictly adhere** to naming conventions- Use the same organizational principles across all projects- Essential for collaboration and team scalability- "If it's not consistent, it's not findable"**2. Clarity Over Cleverness**- Use specific, descriptive folder and file names- Content should be understandable **without opening** the file- Avoid generic terms like "new_folder," "draft1," "temp"- Bad: `doc_v3_final_FINAL.md`- Good: `lyra-cofounder-research_2026-01-23.md`**3. Keep It Shallow (The 3-Click Rule)**- Limit folder depth to **3-4 levels maximum**- Users should reach any file within **3 clicks**- Deep nesting = cognitive overload and wasted time- If you need more than 4 levels, reconsider your categorization**4. Logical Hierarchy**- Organize from broad to specific- Parent folders contain logically grouped subfolders- Mirror mental models (how people naturally think about content)**5. Accessibility & Collaboration**- Structure should facilitate easy sharing- Clear enough for new team members to navigate instantly- Consider using cloud storage (Google Drive, Dropbox, OneDrive) as single source of truth**6. Start Simple, Evolve Gradually**- Don't over-engineer from the start- Begin with basic structure, expand as needed- Overly complex initial structures are hard to maintain**7. Regular Maintenance**- Periodically review and clean up- Archive completed projects- Delete redundant/unnecessary files- Prune ruthlessly**8. Avoid Special Characters**- Stick to: `a-z`, `A-Z`, `0-9`, `.`, `-`, `_`- No spaces (use `_` or `-` instead)- No: `@, #, %, &, *, (, ), <, >`- Prevents compatibility issues across systems---## Popular Organization Systems### 1. The PARA Method (Projects, Areas, Resources, Archives)**Created by:** Tiago Forte (Building a Second Brain)  **Best for:** Personal knowledge management, creative work, research```/├── Projects/              # Active efforts with defined outcomes│   ├── lyra-cofounder/│   ├── website-redesign/│   └── quarterly-report/├── Areas/                 # Ongoing responsibilities (no end date)│   ├── health/│   ├── finances/│   └── professional-development/├── Resources/             # Topics of interest, reference material│   ├── ai-research/│   ├── business-frameworks/│   └── design-inspiration/└── Archives/              # Completed projects, inactive areas    ├── project-x-completed/    └── old-workflows/```**Strengths:**- Action-oriented (Projects first)- Clear lifecycle (Active → Archive)- Easy to maintain- Scales well**When to Use:**- Individual knowledge workers- Creative professionals- Researchers who need fast retrieval---### 2. Johnny Decimal System**Best for:** Structured numbering, large organizations, complex projects```00-09 System/Admin  00 Index  01 Admin  02 Finance10-19 Project Management  10 Planning  11 Execution  12 Reports20-29 Design  20 Concepts  21 Prototypes  22 Final Assets```**Strengths:**- Enforces structure through numbering- Easy to reference: "It's in 21.03"- Hierarchical and scalable- Prevents folder explosion**When to Use:**- Large organizations- Projects with many stakeholders- When precise categorization is critical---### 3. Feature-Based Organization (Monorepo/Software)**Best for:** Software projects, startups, multi-component systems```/├── apps/                  # Deployable applications│   ├── web/│   ├── mobile/│   └── api/├── packages/              # Shared libraries│   ├── ui-components/│   ├── auth/│   └── utils/├── tools/                 # Internal scripts├── configs/               # Shared configs└── docs/                  # Documentation```**Key Principle:** **Group by feature, not by type****Bad (Type-Based):**```/components  - ProfileCard.tsx  - SettingsModal.tsx/services  - profileService.ts  - settingsService.ts```**Good (Feature-Based):**```/profile  - ProfileCard.tsx  - profileService.ts  - profileSlice.ts/settings  - SettingsModal.tsx  - settingsService.ts  - settingsSlice.ts```**Why This Works:**- Everything related to a feature is **co-located**- Easier to understand dependencies- Simpler refactoring- Scales as features grow---## Project-Based Folder Structures### Academic/Research Project Structure**Recommended by:** Harvard, MIT, Duke, McMaster University```project-name/├── 0_admin/                    # They become dumping grounds   - If it doesn't fit, reconsider your categories2. **Use dates in folder names** (unless archiving)   - `/project-2026-01/` becomes confusing   - Use dates in **file names** instead3. **Defer organization**   - "I'll organize it later" → Never happens   - File it right the first time4. **Over-categorize at the start**   - Don't create 50 empty folders "just in case"   - Create as needed5. **Ignore the 3-click rule**   - If navigation takes 5+ clicks, restructure6. **Use ambiguous names**   - "Work stuff" tells you nothing   - Be specific7. **Forget to archive**   - Old projects clutter active workspace   - Move completed work to archives---## Tools & Resources### File Management Tools- **Everything (Windows)**: Instant file search- **Alfred (Mac)**: Powerful file navigation- **fzf**: Command-line fuzzy finder- **Recoll**: Full-text search for documents### Organization Tools- **Obsidian**: Markdown-based knowledge management with linking- **Notion**: All-in-one workspace- **Roam Research**: Networked thought- **Zotero/Mendeley**: Reference management for research### Automation- **Hazel (Mac)**: Automated file organization rules- **PowerShell/Bash scripts**: Custom automation- **Python `os`/`pathlib`**: For complex file operations---## Conclusion**The best folder structure is one you'll actually use.**Key takeaways:1. **Start simple**, evolve as needed2. **Consistency** beats perfection3. **Shallow** hierarchies, descriptive names4. **Maintain** regularly5. **Document** your systemFor Lyra-Cofounder, the recommended numbered structure balances:- Research-first workflow (01_research prominent)- Clear lifecycle (Research → Design → Development)- Scalability (easy to add categories)- Maintainability (intuitive navigation)**Next Steps:**1.
